# SkyAgent 全局记忆与会话记忆 MVP 方案

## 1. 目标

本方案用于为 SkyAgent 增加一套轻量、可控、可演进的“全局记忆 + 工作区记忆 + 会话记忆”能力，让系统在新会话、长会话、文件操作和多轮协作场景下，能够更稳定地理解：

- 当前工作目录是什么
- 这个项目的关键文件和结构是什么
- 用户的长期偏好和当前工作区规则是什么
- 当前会话正在做什么、做到哪一步了

本方案不是要做“无限记忆”，而是做一套 **可解释、可维护、MVP 可落地** 的上下文系统。

---

## 2. 本次 MVP 想解决的问题

当前系统存在以下典型问题：

- 新建会话后，模型对当前项目理解不足，需要重新摸索
- 用户的长期偏好与工作区规则无法稳定复用
- 会话一长，容易出现上下文丢失或执行跑偏
- 文件操作依赖模型临时理解，缺少工作区级别的稳定背景
- 程序缺少一份结构化索引来快速定位关键文件

MVP 目标是先解决“能记住、能复用、能压缩、能解释”这四件事。

---

## 3. 设计原则

### 3.1 分层，不混用

不要把所有信息堆进一个大文件中，而是分成五类：

- 全局长期记忆
- 工作区画像
- 文件索引
- 当前会话上下文
- 用户可见的工作区规则入口

### 3.2 轻量优先

MVP 先记录摘要，不记录全部原文。

- 不把整个目录全文塞给模型
- 不把全部历史会话都塞给模型
- 优先保存“摘要、规则、状态、线索”

### 3.3 程序和模型职责分开

- 给模型读的内容，用 Markdown 摘要文件
- 给程序快速查的内容，用 JSON 结构化索引

### 3.4 可增量更新

所有文件都应支持增量刷新，而不是每次全量重建。

### 3.5 全局记忆要“收紧”

全局记忆必须是一个小而稳的层，只记录跨工作区仍然成立的内容，例如：

- 用户长期偏好
- 稳定回答风格
- 稳定输出风格
- 长期协作习惯

全局记忆不应自动吸收：

- 某个项目特有规则
- 某次会话的临时要求
- 当前目录结构
- 一次性文件操作结论

---

## 4. 记忆分层与职责边界

### 4.1 全局长期记忆

用途：
服务“这个用户本人”，跨项目、跨工作区复用。

职责：

- 记录长期稳定偏好
- 记录跨项目仍成立的协作习惯
- 为新会话提供用户级默认风格

应包含：

- 默认语言
- 回答风格
- UI 偏好
- 文件交付风格
- 稳定工具偏好

不应包含：

- 当前项目目录结构
- 当前项目文件路径
- 某次临时任务
- 某个工作区特有规则

建议形式：

- 一个用户可编辑文件
- 一份系统自动提炼文件
- 一份结构化索引

### 4.2 工作区记忆

用途：
服务“当前工作区/当前项目”。

职责：

- 记录当前工作区下应优先遵守的规则
- 记录当前项目中稳定成立的协作方式
- 记录当前项目的显式约定

应包含：

- 当前项目协作规则
- 当前目录特有约束
- 当前工作区优先遵守的输出方式

不应包含：

- 用户所有长期偏好全文
- 某一轮临时任务目标
- 高频变化的扫描结果

### 4.3 会话记忆

用途：
服务“当前这轮对话”。

职责：

- 记录当前任务目标
- 记录当前涉及文件
- 记录当前已完成事项与待继续事项
- 为长会话压缩提供锚点

应包含：

- 当前目标
- 当前约束
- 当前时间线
- 当前未完成事项

不应包含：

- 全局长期偏好
- 工作区全局规则全文

### 4.4 优先级

记忆冲突时，优先级如下：

1. 当前用户明确指令
2. 当前会话上下文
3. 当前工作区规则
4. 全局长期记忆
5. 默认系统策略

---

## 5. MVP 文件设计

### 5.1 `GLOBAL_SKYAGENT.md`

用途：
作为全局长期记忆的用户可编辑入口文件，供用户直接查看和自定义。

建议位置：

```text
~/.skyagent/GLOBAL_SKYAGENT.md
```

职责：

- 记录跨工作区稳定成立的用户偏好
- 记录用户长期协作习惯
- 作为全局记忆的显式可见入口

典型内容：

- 默认语言
- 回答风格
- UI 风格偏好
- 文件输出习惯
- 代码交付偏好

不应写入：

- 当前项目特有规则
- 某个工作区目录结构
- 某一轮临时任务

### 5.2 `GENERATED_GLOBAL_MEMORY.md`

用途：
系统自动从历史对话中提炼出的全局长期语义记忆。

建议位置：

```text
~/.skyagent/GENERATED_GLOBAL_MEMORY.md
```

职责：

- 自动沉淀跨工作区仍然成立的长期偏好
- 作为 `GLOBAL_SKYAGENT.md` 的补充，而不是替代

### 5.3 `GLOBAL_MEMORY_INDEX.json`

用途：
给程序快速查，用于维护全局长期记忆的结构化索引。

建议位置：

```text
~/.skyagent/GLOBAL_MEMORY_INDEX.json
```

职责：

- 记录全局语义记忆条目
- 支撑检索和去重
- 支撑增量更新

### 5.4 `WORKSPACE_PROFILE.md`

用途：
给模型读，用于描述当前工作区的稳定信息。

建议位置：

```text
~/.skyagent/workspaces/<workspace-id>/WORKSPACE_PROFILE.md
```

职责：

- 描述当前工作目录
- 描述项目类型
- 描述关键目录和关键文件
- 描述项目规则和修改边界
- 描述近期高频模块
- 提供模型进入新会话时的“工作区背景”

典型内容：

- 工作区路径
- 项目类型
- 核心入口文件
- 高优先级目录
- 不建议修改目录
- 最近高频操作文件
- 当前项目约束

不应写入：

- 大量文件正文
- 全量代码
- 长篇日志

---

### 5.5 `FILE_INDEX.json`

用途：
给程序快速查，用于维护工作区中的结构化文件索引。

建议位置：

```text
~/.skyagent/workspaces/<workspace-id>/FILE_INDEX.json
```

职责：

- 记录工作区文件列表
- 标记关键文件
- 提供分类、标签、模块归属
- 支撑程序快速过滤、排序、推荐
- 为模型提供“按需读取”能力，而不是全量读取

典型字段：

- `path`
- `ext`
- `category`
- `tags`
- `isEntry`
- `modifiedAt`
- `size`
- `summary`（可选）

不应写入：

- 大段文件正文
- 临时对话状态

---

### 5.6 `SESSION_CONTEXT.md`

用途：
记录当前会话的阶段性上下文，主要服务长会话与上下文压缩。

建议位置：

```text
~/.skyagent/conversations/<conversation-id>/SESSION_CONTEXT.md
```

职责：

- 记录当前会话目标
- 记录用户已确认要求
- 记录已完成事项
- 记录待继续事项
- 记录当前会话涉及的重要文件
- 在上下文压缩后继续作为会话“摘要锚点”

典型内容：

- 当前任务目标
- 用户明确要求
- 当前涉及文件
- 已完成步骤
- 未完成步骤
- 当前假设
- 当前风险

不应写入：

- 长期偏好
- 全局工作区信息
- 全量聊天原文

---

### 5.7 `SKYAGENT.md`

用途：
作为用户工作区最外层的可见规则文件，记录长期有效的用户偏好、当前工作区规则和协作约定，供模型优先读取。

建议位置：

```text
<workspace-root>/SKYAGENT.md
```

职责：

- 记录用户长期偏好
- 记录当前工作区级别的规则
- 记录稳定的表达风格要求
- 记录稳定的文件处理偏好
- 记录长期确认过的产品/代码习惯
- 作为用户可直接查看和手动调整的入口文件

典型内容：

- 默认语言
- UI 偏好
- 回答风格
- 文件输出偏好
- 常见工作方式
- 持续重复出现的规则
- 当前工作区特殊约束
- 希望优先遵守的协作方式

不应写入：

- 某一轮临时任务
- 某次单独文件改动的细节
- 会频繁波动的扫描结果

---

## 6. 文件之间的关系

### 新建会话时

系统可以按顺序准备上下文：

1. 读取全局 `GLOBAL_SKYAGENT.md`
2. 读取全局 `GENERATED_GLOBAL_MEMORY.md` 的相关摘要
3. 优先读取工作区根目录中的 `SKYAGENT.md`
4. 读取当前工作区的 `WORKSPACE_PROFILE.md`
5. 程序查询 `FILE_INDEX.json`
6. 初始化当前会话的 `SESSION_CONTEXT.md`

### 会话进行中

- `SESSION_CONTEXT.md` 持续更新
- `FILE_INDEX.json` 在文件结构变化后局部更新
- `WORKSPACE_PROFILE.md` 在工作区显著变化后重建或刷新
- `GLOBAL_SKYAGENT.md` 只在长期偏好被确认时更新
- `GENERATED_GLOBAL_MEMORY.md` 只在满足“跨工作区长期有效”时更新
- `SKYAGENT.md` 只在长期偏好或工作区规则被确认时更新

### 会话变长时

系统压缩对话原文，但保留：

- `SESSION_CONTEXT.md`
- 当前活跃文件列表
- 最近关键操作摘要

---

## 7. MVP 的边界

本期不做：

- 不做自动学习所有用户习惯
- 不做隐式长期记忆自动合并
- 不做复杂向量数据库
- 不做全文索引搜索引擎
- 不做全仓库语义图谱
- 不做多工作区关系推理

本期只做：

- 收紧后的全局长期记忆
- 用户可编辑全局记忆文件
- 会话记忆文件骨架
- 新会话初始化
- 长会话压缩承接

---

## 8. 生成与更新时机

### 8.1 `GLOBAL_SKYAGENT.md`

生成时机：

- 首次安装或首次启动，且文件不存在

更新时机：

- 用户明确说出长期偏好
- 用户手动编辑

### 8.2 `GENERATED_GLOBAL_MEMORY.md`

生成时机：

- 第一次从会话中提炼出有效长期记忆

更新时机：

- 对话结束后
- 仅当系统确认内容“跨工作区长期有效”时

### 8.3 `SESSION_CONTEXT.md`

生成时机：

- 新建会话

更新时机：

- 用户目标变化
- 关键文件变化
- 完成一个主要步骤
- 上下文压缩发生时

### 8.4 `WORKSPACE_PROFILE.md`

生成时机：

- 首次进入一个新工作区
- 切换目录后首次打开会话

更新时机：

- 工作区结构发生明显变化
- 用户显式要求“刷新工作区上下文”

### 8.5 `FILE_INDEX.json`

生成时机：

- 首次进入工作区

更新时机：

- 文件创建
- 文件删除
- 文件重命名
- 目录切换

### 8.6 `SKYAGENT.md`

生成时机：

- 首次进入一个工作区，且工作区根目录中不存在 `SKYAGENT.md`

更新时机：

- 用户明确说出长期偏好
- 用户明确给出工作区级规则
- 某项偏好在多个会话中重复出现并被确认

---

## 9. 建议的数据来源

### `GLOBAL_SKYAGENT.md` 来源

- 用户明确表达的长期偏好
- 用户手动维护的全局习惯

### `GENERATED_GLOBAL_MEMORY.md` 来源

- 历史会话中经系统判定为“跨工作区长期有效”的语义信息

### `SESSION_CONTEXT.md` 来源

- 当前会话目标
- 当前会话中的关键工具调用结果
- 当前会话确认过的规则
- 当前 `ConversationContextState`

### `WORKSPACE_PROFILE.md` 来源

- 当前工作目录扫描摘要
- 项目入口文件识别
- 用户显式说明
- 最近实际高频操作文件

### `FILE_INDEX.json` 来源

- 文件系统扫描
- 本地文件操作记录
- 文件意图解析结果

### `SKYAGENT.md` 来源

- 用户明确表达的工作区规则
- 经确认后的协作约定
- 人工确认后沉淀的稳定规则

---

## 10. 模型注入策略

MVP 不建议把所有记忆文件全文都注入模型。

建议注入策略如下：

### 新会话首轮

注入：

- `GLOBAL_SKYAGENT.md` 摘要
- `GENERATED_GLOBAL_MEMORY.md` 相关摘要
- `SKYAGENT.md` 摘要
- `WORKSPACE_PROFILE.md` 摘要

程序内部参考但不全文注入：

- `FILE_INDEX.json`

初始化为空或极小内容：

- `SESSION_CONTEXT.md`

### 会话进行中

优先注入：

- `SESSION_CONTEXT.md`
- 当前相关文件摘要

按需补充：

- `WORKSPACE_PROFILE.md` 中与当前任务相关部分

### 长会话压缩后

保留：

- `SESSION_CONTEXT.md`
- 最近关键操作摘要
- 当前相关文件路径与状态

---

## 11. 推荐的 MVP 实施顺序

### Phase 1：全局记忆与会话记忆

先做：

- `GLOBAL_SKYAGENT.md`
- `GENERATED_GLOBAL_MEMORY.md`
- `SKYAGENT.md`
- `SESSION_CONTEXT.md`

收益：

- 先解决“懂用户”
- 先收紧全局记忆边界
- 先给全局记忆一个用户可编辑入口
- 先提供一个用户肉眼可见、可编辑的工作区入口
- 先解决“长会话不丢事”

### Phase 2：工作区画像

再做：

- `WORKSPACE_PROFILE.md`

收益：

- 解决新会话对项目不熟的问题

### Phase 3：文件索引

最后做：

- `FILE_INDEX.json`

收益：

- 提升程序检索效率
- 为更智能的文件意图处理打基础

---

## 12. MVP 验收标准

满足以下条件即可认为 MVP 可用：

- 全局长期偏好可通过 `GLOBAL_SKYAGENT.md` 显式查看和修改
- A 会话中的一次性信息不会轻易污染 B 会话的全局记忆
- 新建会话后，模型能知道当前工作区的基本类型和关键入口
- 工作区根目录中的 `SKYAGENT.md` 可被新会话稳定复用
- 长会话压缩后，任务目标和已完成事项不会明显丢失
- 文件操作前，程序能够快速拿到候选关键文件
- 所有记忆文件都可解释、可查看、可手动调试

---

## 13. 风险与注意事项

### 12.1 信息过期

风险：
工作区结构变化后，画像和索引可能过期。

应对：

- 加时间戳
- 支持增量刷新
- 提供“刷新工作区上下文”入口

### 13.2 信息膨胀

风险：
会话长了以后，`SESSION_CONTEXT.md` 可能变成流水账。

应对：

- 限制长度
- 只保留任务级摘要
- 定期压缩覆盖

### 13.3 长期记忆污染

风险：
一次性要求被错误写入全局记忆或工作区记忆

应对：

- 全局记忆仅记录跨工作区稳定成立的内容
- 工作区记忆只记录项目级稳定规则
- 优先人工确认或多次重复出现后再沉淀

### 13.4 模型依赖记忆而不读文件

风险：
模型误把记忆当事实源，不再读取真实文件。

应对：

- 明确规定：记忆只提供线索，真实内容仍以文件读取结果为准

---

## 14. 建议的最终结论

本方案建议执行，但按 MVP 顺序逐步落地：

1. 先做 `GLOBAL_SKYAGENT.md`
2. 再做 `SESSION_CONTEXT.md`
3. 再做 `SKYAGENT.md`
4. 再做 `WORKSPACE_PROFILE.md`
5. 最后做 `FILE_INDEX.json`

这样可以用最小成本，先得到最明显的体验提升。

如果后续继续演进，这套结构也能自然扩展到：

- 更精确的上下文压缩
- 更智能的文件意图识别
- 多工作区管理
- 更强的工具/文件协作能力
